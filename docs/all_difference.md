## 一、架构差异

### 1. GCN 层数
- **论文**：≥2 层，标准多层传播 `H^{(l+1)} = σ(ÃH^{(l)}W^{(l)})` (Eq. 8-9)
- **代码**：[dgcgru.py:110-114](REGCN/dgcgru.py#L110-L114) 仅 **1 层** GCN，无归一化、无层间激活
- **状态**：✅ 已实现（单层）
- **理由**：因图只有 6-9 个节点且全连接，单层已让所有节点互通，多层反而易过平滑。是合理的工程简化。

### 2. Multi-GCN 图融合方式
- **论文**：标量加权求和 `A_s = Σ w_k A_k` (Eq. 7)
- **代码**：[dgcgru.py:58-64](REGCN/dgcgru.py#L58-L64) **逐边可学习权重** `A_s = Σ W_k ⊙ A_k`，每个元素独立学习，参数量 `3N²` 而非 3
- **状态**：✅ 已实现（per-edge 注意力增强）
- **理由**：代码实现灵活性高于论文，可视为扩展增强。

### 3. GCN-GRU 结合方式
- **论文**：两阶段流水线，GCN 先生成 node embedding，再送入 GRU
- **代码**：[dgcgru.py:91-108](REGCN/dgcgru.py#L91-L108) GCGRU cell，GCN **嵌入在 GRU cell 内部**，每时间步先 GCN 再进入门控
- **状态**：✅ 已实现

### 4. GRU 替代 LSTM 用法的合理性
- **论文**：Section 3.6 明确使用 GRU，代码一致
- **状态**：✅ 一致

## 二、损失函数差异（核心差异，曾导致模型塌缩）

### 5. 论文公式 vs 代码实现对比

**论文 Eq. 14-15：**
```
L = λ₁ · (1/n) Σ(y - ŷ)²   +   λ₂ · (1/n) Σ 1[sign(Δy) ≠ sign(Δŷ)]
    ↑  λ₁ 直接乘在 MSE 项上        ↑  趋势方向惩罚
      （论文默认 λ₁=0.01）                （论文默认 λ₂=0.1）
```

**代码实际结构：**
```
loss = MSE + r_acc · trend_penalty         ← [losses.py:32] MSE 权重恒为 1
       + r_mse · ‖W_dense‖₁                ← [REGCN.py:105] Dense 层的 L1 权重衰减
```

### 6. 两处关键差异

**差异 A：λ₁ 的物理含义不同**

| | 论文 | 代码 |
|---|---|---|
| λ₁ 作用对象 | MSE 项本身 | Dense(1) 输出层的权重（128 个参数） |
| 数学形式 | λ₁ · (1/n)Σ(y-ŷ)² | r_mse · Σ\|w\|（L1 权重衰减） |
| 效果 | 缩放 MSE 梯度 | 迫使输出层权重稀疏 |
| 默认值 | 0.01 | 0（关闭） |

**差异 B：历史 Bug — L1 曾被错误应用到 GCGRU 循环权重（已修复）**

在较早版本中，`r_mse` 不仅作用于 Dense 层，还通过 `kernel_regularizer` 参数传入 GCGRU cell，被应用到 **所有 9 个循环权重矩阵**（`wz, wr, wh, w0, wa, uz, ur, uh`，见 [dgcgru.py:37-80](REGCN/dgcgru.py#L37-L80)）：

```
受影响参数量（L1 全开时）：
  Dense(1):       128 个权重 ✓ 可以接受
  + GCGRU cell:   ~9 × 128² = 147,456 个权重 ✗ 毁灭性
```

当 `r_mse=0.01` 时，L1 惩罚覆盖 14 万个参数，优化器发现**把所有权重压到零、通过无正则化的偏置（`bz, br, bh`）输出常数**更"便宜"。这导致模型对所有测试样本输出完全相同数值。

**验证实验（来自代码审查）：**

| 配置 | R² | 预测值范围 | 说明 |
|:----|:--:|:----------:|------|
| L1 全开 (`r_mse=0.01`, GCGRU + Dense) | **-0.017** | 常数（4.715） | 模型塌缩为常数 |
| L1 关闭 (`r_mse=0.0`) | **+0.588** | 有变化（0.34 范围） | 恢复正常预测 |

修复后（当前代码 [REGCN.py:101](REGCN/REGCN.py#L101)），GCGRU cell **不再接收** `kernel_regularizer`，L1 仅作用于 Dense 输出层的 128 个权重。`config.ini` 默认 `r_mse=0` 使 L1 完全关闭。

### 7. 趋势正则化项的差异

| | 论文 | 代码 |
|---|---|---|
| 趋势项形式 | 一致：`(1/n) Σ 1[sign(Δy) ≠ sign(Δŷ)]` | ✅ 一致 |
| 加权方式 | λ₂ 乘在趋势项上 | `r_acc` 乘在趋势项上 |
| 默认值 | 0.1 | 0（关闭） |

趋势正则化本身实现正确 ([losses.py:20-30](REGCN/losses.py#L20-L30))，但默认关闭。

### 8. 配置建议

如要按论文参数运行，需手动设置：
```ini
r_mse = 0.0001      # 论文 0.01 对 Dense 层偏大（实测 r_mse=0.0001 最优，见图4）
r_acc = 0.1         # 与论文一致
```

注意：即使设了论文参数，代码的 `r_mse=0.01` 是 L1 权重衰减而非 MSE 缩放，与论文 λ₁ 物理含义不同。Grid Scan（Figure 4）结果显示 `r_mse=0.0001, r_acc=0` 在 Stock 0 上取得最佳 R²=0.96。

## 七、复现差距总结

### 7.1 数值对比

论文声称的指标（Table 5）：
| 数据集 | ACC | R² | RMSE | MAE | MAPE |
|:------|:---:|:--:|:----:|:---:|:----:|
| SSE | 0.6940 | 0.9226 | 0.1500 | 0.1137 | 0.843% |
| DJIA | 0.7311 | 0.9652 | 1.6082 | 1.2206 | 0.714% |

实际复现全量均值（2026-06-19）：
| 数据集 | ACC 均值 | R² 均值 | R² 中位 | RMSE | 正 R² 比例 |
|:------|:-------:|:-------:|:-------:|:----:|:---------:|
| SSE | 0.6084 | 0.6688 | 0.8843 | 0.4570 | 37/40 (92.5%) |
| DJIA | 0.5888 | 0.7074 | 0.8559 | 4.4455 | 27/28 (96.4%) |

### 7.2 差距原因分析

1. **均值被少数灾难性股票严重拖低**：SSE stock 29 在某些配置下 R²=-58.6，Stock 21 的 R²=-0.83。这些极端负值大幅拉低均值，但 **中位 R² 约 0.86-0.88** 显著更接近论文值（0.92/0.97）。论文很可能报告的是中位或剔除了异常股票后的均值。

2. **架构差异**（见第一节）：
   - 单层 GCN vs 论文的多层 → 表达能力略降
   - Per-edge 可学习权重 vs 标量加权 → 灵活性更高但参数量大
   - 逐窗口归一化 vs 全局归一化 → 分布更稳定但信息损失

3. **论文对超参数的默认描述与最优值不符**：
   - 论文声称 `λ₁=0.01`（MSE 缩放系数），但代码中 `r_mse=0.01` 是 L1 权重衰减，作用在 128 个 Dense 权重上而非 MSE 项
   - Grid Scan（Figure 4）显示 `r_mse=0.0001, r_acc=0` 在 Stock 0 上取得 R²=0.9607，远优于论文默认值
   - 论文的 `λ₁=0.01` 若按代码语义（L1 权重衰减）会导致输出层被过度稀疏化

4. **随机种子未固定**：TF 的随机初始化、数据洗牌、Dropout（若有）均未固定种子，每次运行结果有浮动。两个 seq_len=30 的独立运行间，同一股票 R² 平均绝对差可达 1.25（受少数灾难性股票影响）。

## 八、运行过程与工程决策

### 8.1 计算环境选择

| 环境 | 规格 | 用途 |
|:----|:----|:-----|
| **本地 Windows** | i7-12700H, 16GB, TF 2.21 CPU | 代码开发、单支股票调试、Figure 生成 |
| **DeepLN 云主机** | AMD EPYC 7H12 × 2 (256 vCPU), 503GB RAM, TF 2.12 CPU | 全量实验（SSE 40 + DJIA 28） |

**为什么不用 GPU？**
- 本地显卡（RTX 3060 Max-Q 6GB）不兼容 TF 2.21（Windows 上 TF ≥ 2.11 不支持 CUDA）
- 云主机无 GPU，但有 256 核 CPU，TF 通过 oneDNN 利用 CPU 并行计算
- 单支股票 100 epoch 训练约 50-70s（本地）/ 30-60s（云端，intra=16）

### 8.2 云端并行策略

由于只有 24 核可实际使用（非独占整机），采用以下策略：
- **串行**：n_jobs=1，1 支/次，利用 16 TF intra-op threads → SSE 40 支约 40min
- **并行**：--workers=16，16 支同时训练，利用 multiprocessing fork → SSE 40 支约 3min
- **注意**：6 个消融方法 × 16 workers 同时启动 = 96 进程会打崩 SSH 服务，需串行化方法队列

### 8.3 遇到的特别困难

**(1) L1 正则化陷阱（已修复）**
论文将 `kernel_regularizer=l1(0.01)` 应用于 GCGRU cell 的全部 9 个循环权重矩阵（约 98K 参数），导致 L1 惩罚比 MSE 大 500 倍以上。优化器发现"把所有权重压到零，通过偏置输出常数"更便宜。模型对所有测试样本输出相同数值（R²≈0 甚至是负值）。
- **修复**：将 L1 从 GCGRU cell 移除，仅保留在 Dense(1) 输出层的 128 个权重上。config.ini 中 `r_mse=0.0` 使 L1 完全关闭。

**(2) TF 2.21 / Keras 3 兼容性**
本地环境（TF 2.21 / Keras 3.13）保存的权重格式与云端（TF 2.12 / Keras 2.12）不兼容：
- save_weights() 在 Keras 3 用变量路径（`layers/rnn/cell/vars/0`）而非变量名
- load_weights() 需要完全相同版本的 Keras
- 跨环境加载权重导致 R² 从 0.91 骤降到 -126
- **解决**：分开跑，各自训练各自预测

**(3) Multiprocessing + TF 冲突**
TF 的全局状态与 multiprocessing 的 spawn 上下文冲突：
- `spawn` 进程重复初始化 TF 导致死锁或 10+ 分钟初始化
- Python 的 `multiprocessing.get_context('spawn')` 在 Linux 上改用 `fork` 后稳定
- Windows 上只能用 `spawn`，但 4 进程以内工作正常

**(4) 云端文件路径中文乱码**
本地项目路径包含中文字符，h5py 在 Windows 上无法打开包含中文路径的 `.h5` 权重文件（GBK 编码限制）。
- **解决**：将模型文件复制到纯英文路径（`C:\temp\stock_models\`），或直接在 Linux 云端运行

**(5) 云端 SSH 被打死**
6 个消融方法 × 16 workers 同时启动 = 96 进程同时加载 TF，SSH 服务因资源耗尽断开连接。
- **解决**：一次只跑一个方法，最多 16 workers

### 8.4 代码不修改原始文件的原则

所有消融实验（Table 6）和辅助工具（Figure 生成等）均通过**外部脚本**控制，不修改 REGCN/ 下的原始代码：
- `table6_ablation.py` — 消融实验运行器，通过参数注入控制 VMD 循环 / 邻接矩阵 / 损失函数
- `fig3_prediction_plots.py` — 从已保存的权重文件直接预测并生成对比图
- `fig4_regularization_scan.py` — 正则化系数 Grid Scan
